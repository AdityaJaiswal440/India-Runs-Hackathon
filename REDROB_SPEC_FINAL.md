## SPEC-0 —System Mental Model

### 0.1 Fundamental Premise

This system is NOT a classifier. It reverse-engineers a hidden ground-truth scoring function. That function was generated programmatically from the same data available here: static profile + 23 behavioral signals. NDCG measures alignment between this system's output ranking and the hidden ground truth.

The v2 system described in the JD is the minimum target. This system builds v3.

### 0.2 JD Intelligence Mapping

The job_description.docx is a dual-purpose document encoding the ground-truth scoring function architecture:

| JD Language                                                  | System Architecture Implication                                                                                              |
| ------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------- |
| "mostly BM25 + rule-based scoring, working but not great"    | v1 baseline. Ground truth generated by v2+. This system must exceed v2.                                                      |
| "v2 with embeddings, hybrid retrieval, LLM-based re-ranking" | Ground-truth architecture this system must match and exceed.                                                                 |
| "recruiter-engagement metrics"                               | Target signals:`saved_by_recruiters_30d`, `profile_views_received_30d`, `search_appearance_30d`. Must NOT be excluded. |
| "offline benchmarks, online A/B testing"                     | NDCG@10/NDCG@100 evaluation + submission validator IS this infrastructure.                                                   |
| "candidate-JD matching at scale"                             | 100K candidates, CPU-only, 5 minutes. Production constraints.                                                                |

### 0.3 Four Architectural Axioms

**Axiom 1 — RRF not linear sum:** Fusion mechanism is Reciprocal Rank Fusion (k=60), not a linear weighted sum. RRF is scale-invariant, parameter-free, consistently outperforms fixed-weight combinations.

**Axiom 2 — HyDE not raw JD embedding:** Dense retrieval uses a Hypothetical Resume Embedding. CONFIT V2 (ACL Findings 2025) demonstrates +22% nDCG from embedding an LLM-generated ideal candidate profile rather than the JD text directly.

**Axiom 3 — `search_appearance_30d` is mandatory:** Direct recruiter-engagement signal. Exclusion violates the JD's stated v2 optimization target.

**Axiom 4 —**

> ** LLM used exactly once, offline only:** A Large Language Model is used exactly once in the entire project — Gate P2, before `precompute.py` runs, to generate the hypothetical JD resume. This is a manual, offline step performed using the free Claude web UI (or any free LLM web UI). The output is saved to a text file. All ranking and reasoning logic in the codebase is procedural, deterministic, and network-free. The submission spec explicitly bans any API/network calls during `rank.py` execution ("Network Off — no OpenAI, Anthropic, Cohere, Gemini, or any hosted LLM service").

### 0.4 Three Built-In Dataset Traps

## SPEC-1 — Scoring Math: Where to Spend Engineering Hours

### 1.1 Official Scoring Formula

```
Final Composite = 0.50 × NDCG@10 + 0.30 × NDCG@50 + 0.15 × MAP + 0.05 × P@10
```

| Metric  | Weight         | Implication                                              |
| ------- | -------------- | -------------------------------------------------------- |
| NDCG@10 | **0.50** | Top 10 picks alone = half your score. Obsess over these. |
| NDCG@50 | 0.30           | Top 50 quality matters significantly                     |
| MAP     | 0.15           | Precision across all relevance levels                    |
| P@10    | 0.05           | Fraction of top-10 that are "relevant" (tier 3+)         |

**Strategic directive:** A single bad candidate in your top 10 — a keyword stuffer, a domain mismatch, an inactive ghost profile — costs ~5% of your total score, larger than most weight-tuning gains. Spend 80% of calibration effort on ranks 1-10.

Tiebreakers (if scores tied): Higher P@5 → Higher P@10 → Earlier submission timestamp.

### 1.2 Honeypot Disqualification Threshold

The dataset contains approximately 80 honeypot profiles across 100K candidates. If more than 10 honeypots appear in your submitted top 100 (>10%), this is an **instant Stage 3 disqualification** before ranking quality is ever scored.

After your first rank.py run, manually inspect every candidate in your top 100. More than 5 structurally-impossible profiles → emergency: strengthen is_honeypot() rules immediately.

### 1.3 Three Submissions — Last One Counts

Maximum 3 submission attempts. The **last valid submission counts** — not the highest-scoring one. A weaker final submission overwrites a stronger earlier one.

```
Final Composite = 0.50 × NDCG@10 + 0.30 × NDCG@50 + 0.15 × MAP + 0.05 × P@10
```

## SPEC-2 — Determinism Anchor (`src/constants.py`)

### 2.1 The Problem

`behavioral_multiplier()`, `concern_fact()`, and `is_honeypot()` all compute `days_inactive`. If "today" is `date.today()`, code produces different rankings depending on what day it runs. Stage 3 evaluators run your code weeks later, possibly on a hidden/refreshed dataset. A naively hardcoded date (e.g., a fixed calendar date chosen at build time) fails for the opposite reason: once real-world evaluation happens *after* that hardcoded date, every candidate's `days_inactive` becomes inflated by the gap between the hardcoded date and the actual evaluation date — collapsing the entire behavioral signal into noise (most candidates hit the harshest `>180 days` penalty tier regardless of their actual recency) and creating exactly the kind of silent miscalibration this spec exists to prevent.

> ⚠️ **AUDIT FIX (Issue 1 — corrected, not adopted as proposed):** An external audit of this document proposed solving the determinism problem by hardcoding `TODAY = date(2026, 6, 28)` in `src/constants.py`, reasoning that a fixed, known-in-advance date "produces identical output on every machine forever." This reasoning is incomplete: identical output forever is only achieved if the hardcoded date is in the *future* relative to every possible evaluation run — which cannot be guaranteed, since Stage 3 may run this pipeline at any point after submission, including potentially weeks past a hardcoded deadline date. The moment evaluation crosses that hardcoded date, `days_inactive` for the *entire pool* shifts upward by the same fixed offset, and most candidates collapse into the `>180 days → ×0.40` penalty tier together — destroying the behavioral signal's discriminative power exactly when Stage 3 is reading it. The fix below resolves the *actual* problem (non-determinism across repeated runs of the *same* dataset) without introducing this new failure mode.

### 2.2 The Fix: Dataset-Derived Temporal Anchor (`DATASET_TODAY`)

**Core principle:** Determinism does not require a date that never changes — it requires a date that is a pure function of the input data. Two runs of `rank.py` against the *identical* `candidates.jsonl`/`candidates.jsonl.gz` file must always produce the *identical* `TODAY`, regardless of which calendar day either run actually executes. If the underlying dataset genuinely changes (e.g., Stage 3 evaluates against a refreshed or hidden pool with different `last_active_date` values), `TODAY` is *expected* to shift — because "today," for the purposes of measuring recency, is properly defined relative to that dataset's own timeline, not relative to an arbitrary fixed point unrelated to the data being scored.

**Definition:** `DATASET_TODAY` is the most recent `last_active_date` across the entire candidate pool, clamped so that a corrupted or honeypot-style future-dated record cannot distort the anchor for everyone else.

**In `precompute.py` (runs first — computes and persists the anchor):**

```
Final Composite = 0.50 × NDCG@10 + 0.30 × NDCG@50 + 0.15 × MAP + 0.05 × P@10
```

**In `src/constants.py` (loaded by both `precompute.py` and `rank.py`):**

```
Final Composite = 0.50 × NDCG@10 + 0.30 × NDCG@50 + 0.15 × MAP + 0.05 × P@10
```

**Why this satisfies `deterministic_mode: true` (SPEC-19) correctly:**

- Re-running `precompute.py` on the identical `candidates.jsonl` produces the identical `DATASET_TODAY`, every time, on every machine, forever — the actual guarantee `deterministic_mode: true` requires.
- Re-running on a genuinely different dataset (Stage 3's hidden pool) correctly produces a *different* `DATASET_TODAY`, because that pool has a different real "most recent activity" reference point. This is not a determinism violation — it is the correct behavior. Determinism means *reproducibility for a given input*, not *invariance across different inputs*.
- `artifacts/dataset_today.txt` is committed to git (a single ISO date string, a few bytes) so a reviewer auditing the repository sees exactly which anchor date was used for the team's own development and calibration. Stage 3 regenerates its own value from its own dataset during the Docker `precompute.py` run, as expected by SPEC-11's "evaluators run precompute.py first" sequence.
- No hardcoded date exists anywhere in the codebase to go stale, require updating before a deadline, or silently corrupt scoring if forgotten.

**Guard against corruption (already embedded above):** A malformed or maliciously future-dated `last_active_date` (e.g., a Module 5.14 Rule 21/35-style honeypot trap with a future date) cannot shift the anchor, because `compute_dataset_today()` clamps every candidate date against `real_world_today` before taking the max.

### GATE P1 — Data Distribution Audit

**Timing:** The moment you unzip the data. Before writing a single weight.

**Why non-negotiable:** Every weight is calibrated against assumptions from one sample candidate. If 60% of the real pool is consulting-only backgrounds, your company_type_score is crushing the majority before the behavioral multiplier even runs. Takes 2 minutes. Changes everything downstream.

**Logic Specification:**

Stream-parse `candidates.jsonl` line-by-line (never load entire file into memory). Accumulate:

- `all_consulting`: candidates where EVERY job in career_history is at a consulting firm (word-boundary match against {tcs, infosys, wipro, accenture, cognizant, capgemini, hcl technologies, ltimindtree})
- `open_to_work`: count where `open_to_work_flag == True`
- `no_github`: count where `github_activity_score == -1`
- YoE buckets: 2-year ranges
- Location tiers: tier1={pune, noida, delhi, gurgaon} substring; tier2={bangalore, mumbai, hyderabad}; other
- **Optional-field sparsity:** count skills missing `duration_months`; count education entries missing `tier` — these are OPTIONAL fields per candidate_schema.json; knowing their missingness rate tells you how often .get() defaults are actually firing

**Decision Table:**

| Observation                            | Calibration Action                                                                                                                                                                                    |
| -------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `all_consulting` > 40%               | Loosen company_type_score floor from 0.02 to 0.05                                                                                                                                                     |
| `open_to_work=false` > 60%           | Relax open_to_work_flag=False multiplier from ×0.80 to ×0.90                                                                                                                                        |
| Location tier1 < 5%                    | Raise location weight 0.07→0.10; trim education weight 0.03→0.00                                                                                                                                    |
| YoE skewed toward 3-5y                 | Verify experience_fit parabola does not over-penalize dominant bucket                                                                                                                                 |
| >30% skills missing`duration_months` | Duration-trust formula zeroes missing skills; large gap means skill_match_score under-rewards incomplete-but-real candidates; confirms 0.22 skill_trust weight is correctly calibrated conservatively |
| >30% education entries missing`tier` | education_score defaults to "unknown" (0.40) for these; confirms 0.03 weight is appropriately small                                                                                                   |

**Output:** Printed distribution summary. No artifacts generated.

### GATE P2 — LLM-Generated Hypothetical JD Resume (Proper HyDE)

**Timing:** Before running precompute.py. One-time, offline only.

**Why non-negotiable:** `candidate_embeddings.npy` and RRF scores are computed once. A mediocre hypothetical resume baked in cannot be fixed without rerunning the full 20-35 minute precompute. The API call costs nothing — one prompt, three variations, pick the best. This also enables the Stage 5 claim: "We implemented HyDE properly (LLM-generated), not a static approximation."

**This is a manual, offline step. No API key or paid subscription is required.**

**Logic Specification:**

1. Open your free Claude (or ChatGPT/Perplexity) web UI.
2. Copy and paste the **HyDE Generation Prompt** (below) into the chat.
3. Paste the *entire* text of `job_description.docx` (or `.md`) into the `[INSERT JOB DESCRIPTION HERE]` placeholder at the bottom of the prompt.
4. Run the prompt 3 times to get 3 distinct variations. Save each variation locally (e.g., `var1.txt`, `var2.txt`, `var3.txt`).

**The HyDE Generation Prompt (Dynamic Template):**

```
Final Composite = 0.50 × NDCG@10 + 0.30 × NDCG@50 + 0.15 × MAP + 0.05 × P@10
```

### Why this prompt is mathematically superior for your pipeline:

1. **First-Person Constraint:** The candidate corpus contains first-person profile summaries (`"Software / data professional with 6.9 years..."`). Forcing the HyDE resume to be first-person aligns the embedding vectors perfectly in the same semantic space.
2. **Structural Mimicry:** By forcing a "Headline + Summary + Experience" format, the generated text structurally matches the `build_candidate_text()` function in Module 5.5, ensuring the cosine similarity computation is comparing apples to apples.
3. **Exact Terminology Rule:** Instructs the LLM to explicitly use the JD's exact phrases (e.g., "hybrid retrieval", "BM25"). This ensures the dense embedding vector lands in the exact same cluster region as candidates who actually have those skills.
4. **Dynamic by Design:** If the evaluators swap the JD in Stage 3, you can simply paste the new JD into the placeholder, and the prompt will dynamically generate a perfect resume for the new role on the fly.
5. Selection protocol:

   - Source 10 known-good candidates from the **FULL `candidates.jsonl` (100K rows) via grep** — NOT from `sample_candidates.json`. The sample file is a small curated demo bundle, not representative of the real population's vocabulary, phrasing, or skill distribution. Selecting from the full dataset means the HyDE resume is validated against the actual scoring distribution.
   - Command: `grep -i '"current_title".*"ML Engineer"' data/candidates.jsonl | head -5 > /tmp/pool.jsonl`; add Search Engineers similarly; manually inspect and keep 10 genuinely strong fits (retrieval skills, duration_months>12, product-company career)
   - Embed the 10 known-good candidates using `all-MiniLM-L6-v2` with L2-normalization
   - Embed each of 3 hypothetical resume variations with L2-normalization
   - For each variation, compute average cosine similarity (dot product of L2-normalized vectors) against all 10 known-good embeddings
   - Select variation with highest average cosine similarity
6. Save selected text to `hypothetical_resume_selected.txt`
7. **Commit `hypothetical_resume_selected.txt` to git IMMEDIATELY.** precompute.py reads this file. Missing it on a fresh clone → FileNotFoundError → Stage 3 Docker crash → disqualification.

**Boundary Conditions:**

- Runtime: < 2 seconds for selection step
- No circular dependency on precompute.py output — selection uses 10 manually-sourced candidates
- `hypothetical_resume_selected.txt` MUST be in git. It is NOT in .gitignore.

### GATE P3 — Weight Calibration (Numerical, Not Intuitive)

**Timing:** 2-3 hours after first precompute.py completes.

**Why non-negotiable:** Weight calibration cannot happen until you have real scores. Most teams calibrate by eye. This plan uses numerical sensitivity testing — every weight perturbation measured, overlap quantified, stability confirmed.

**Calibration Gate Checklist (run after EVERY rank.py execution):**

```
Final Composite = 0.50 × NDCG@10 + 0.30 × NDCG@50 + 0.15 × MAP + 0.05 × P@10
```

**Sensitivity Test Logic:**

- Pre-compute behavioral multipliers once (identical across all perturbations)
- For each of 8 weight dimensions: perturb by +0.05 and -0.05
- Renormalize perturbed weights to sum 1.0
- Compute top-10 using: `base_scores = features[:, :8] @ perturbed_weights`, then `final_scores = base_scores * multipliers`, then `final_scores[honeypot_mask] = 0.0`, then argsort descending
- Measure overlap between perturbed top-10 and base top-10
- **Pass criterion:** All 8 dimensions ≥8/10 overlap
- **Action threshold:** Any dimension <7/10 → over-weighted → trim it
- Sensitivity test uses `behavioral_multiplier(candidate_store[cid])` — the **FULL candidate dict**, not just `redrob_signals` (required because behavioral_multiplier also reads `current_title` from profile for the architect-stopped-coding check; see SPEC-5 Module 5.13.3)

**Failure Mode → Fix Table:**

| Symptom                                    | Root Cause                     | Exact Fix                                           |
| ------------------------------------------ | ------------------------------ | --------------------------------------------------- |
| Consulting firms in top 10                 | career weight too low          | career 0.20→0.26, text_retrieval 0.30→0.24        |
| Inactive candidates in top 10              | `last_active_date` parse bug | Check date.fromisoformat() — not a weight issue    |
| Keyword stuffers in top 10                 | skill_trust weight too low     | skill_trust 0.22→0.28, text_retrieval 0.30→0.24   |
| Good text, empty skill arrays ranking high | skill_coverage weight too low  | skill_coverage 0.06→0.10, engagement 0.02→0.00    |
| Strong ML candidates NOT in top 100        | text_retrieval weight too low  | text_retrieval 0.30→0.36, location 0.07→0.04      |
| Top 10 all from same location              | Token-split location bug       | Debug location_tokens re.split() output for samples |

**After every weight change: re-run the weight sum assertion. Never skip it.**

### GATE P4 — Anti-Hallucination Test

**Timing:** Before every submission. No exceptions.

**Why non-negotiable:** Stage 4 auditors sample 10 **random** rows from the full 100 rows — not just your top 20. A hallucination at rank 73 fails Stage 4 as badly as one at rank 2. This script costs <1 second to run against all 100 rows.

**Logic Specification:**

## SPEC-4 — Stage 1 Validator: Six Hidden Traps

> These six traps cause instant Stage 1 elimination before ranking quality is ever scored. Understand them before writing rank.py's CSV output logic.

| Trap                       | Description                                                                                                                                                                        | Fix                                                                                                                   |
| -------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| Trap 1: Tie-break enforced | Validator explicitly checks: if score_A == score_B and candidate_id_A > candidate_id_B → ERROR.                                                                                   | Sort by`(-score, candidate_id)` BEFORE assigning ranks. Never sort by ID after rank assignment.                     |
| 2: Float precision         | Validator parses scores back to floats and does exact equality comparisons. Scores that round to the same 4-decimal string are treated as equal and trigger tie-break enforcement. | Write scores with 6 decimal places:`f"{score:.6f}"`. Rely on Trap 1 pre-sort as the tie-break, not float precision. |
| 3: Integer rank parsing    | Validator does`int(rank_s)` then checks `str(rank) == rank_s`. "01" fails (str(1)=="1", not "01"). "1.0" fails.                                                                | Always cast rank to plain Python`int`: `writer.writerow([cand_id, int(rank), f"{score:.6f}", reasoning])`         |
| Trap 4: UTF-8 BOM          | `encoding="utf-8-sig"` prepends `\ufeff` to first cell, silently turning "candidate_id" into "\ufeffcandidate_id". Exact header match fails.                                   | Always`encoding="utf-8"`. Never `utf-8-sig`.                                                                      |
| Trap 5: Hand-rolled CSV    | Raw string concatenation (f.write(f"{a},{b},{c}\n")) breaks on reasoning text containing commas. Validator checks column count.                                                    | Always use`csv.writer` — automatically quotes fields containing commas.                                            |
| Trap 6: Silent NaN         | `float("nan")` parses successfully. NaN breaks all comparisons and corrupts monotonicity check silently.                                                                         | Add before argsort:`assert not np.any(np.isnan(final_scores))` and `assert not np.any(np.isinf(final_scores))`    |

### Module 5.2 — Per-Skill Trust Score (Anti-Stuffing)

**Output:** `skill_match_score` ∈ [0.0, 1.0] → features.npy column [1]

**Mathematical Logic (per skill in candidate["skills"]):**

1. **Lookup:** `base_weight = JD_TAXONOMY[s.name.lower().strip()]`. If not in taxonomy → skip (contributes 0, no penalty).
2. **Proficiency factor:** beginner→0.30, intermediate→0.60, advanced→0.85, expert→1.00. Missing/unrecognized→0.30.
3. **Duration gate:** If `s.duration_months == 0` OR field is missing (default 0) → **skip entirely**. Zero-duration skills contribute nothing regardless of claimed proficiency.
4. **Duration trust:** `min(s.duration_months / 24.0, 1.0)` — saturates at 1.0 for 24+ months.
5. **Assessment factor:** If `s.name.lower()` exists in `candidate["redrob_signals"]["skill_assessment_scores"]` → `score / 100.0`. Otherwise → 0.65 (mild penalty for unverified).
   - ⚠️ Assessment scores are in `redrob_signals.skill_assessment_scores` (NOT in the skills array itself).
6. **Endorsement factor:** `min(s.endorsements / 25.0, 1.0)` — saturates at 25+ endorsements.
7. **Per-skill contribution:**
   ```
   Final Composite = 0.50 × NDCG@10 + 0.30 × NDCG@50 + 0.15 × MAP + 0.05 × P@10
   ```
8. **Aggregate and normalize:**
   ```
   Final Composite = 0.50 × NDCG@10 + 0.30 × NDCG@50 + 0.15 × MAP + 0.05 × P@10
   ```

**Post-computation multiplier (apply immediately after):**

```
Final Composite = 0.50 × NDCG@10 + 0.30 × NDCG@50 + 0.15 × MAP + 0.05 × P@10
```

**Boundary conditions:**

- Skills with missing `duration_months` default to 0 → skipped by duration gate
- Skills not in JD_TAXONOMY silently ignored
- Final score clipped to [0.0, 1.0]

**Data contract:**

- Input: `candidate["skills"]`, `candidate["redrob_signals"]["skill_assessment_scores"]`
- Output: `skill_match_score` ∈ [0.0, 1.0] (after LangChain penalty), stored column [1]

### Module 5.3 — BM25 Text Retrieval

**Output:** `bm25_scores_norm` ∈ [0.0, 1.0] → input to RRF fusion (Module 5.5)

**Logic Specification:**

1. **Corpus construction:** For each candidate, concatenate `profile.summary` + all `career_history[].description`. Tokenize by lowercasing and splitting on whitespace.
2. **Vocabulary pruning (CRITICAL for disk budget):**

   - Compute document frequency for every token across all 100K documents
   - Remove hapax legomena (tokens in only 1 document) — retain tokens with doc frequency ≥ 2
   - If bm25_model.pkl exceeds 1 GB after pruning: increase floor to doc frequency ≥ 3
   - Rationale: single-document terms have near-zero IDF and cannot match the query; pruning reduces pickle size 60-70%
3. **Index construction:** `BM25Okapi(pruned_corpus)`
4. **Query tokens:**

   ```
   Final Composite = 0.50 × NDCG@10 + 0.30 × NDCG@50 + 0.15 × MAP + 0.05 × P@10
   ```

   ⚠️ Multi-word strings like "hr tech" or "hr-tech" are NOT valid BM25 query tokens. BM25Okapi does token-level lookup; a corpus tokenized by whitespace-split will never match a multi-word query string as a single token. Only single-word tokens are valid here. "hr" and "tech" can appear as separate single-word tokens if needed.
5. **Score computation:** `bm25_raw = bm25.get_scores(JD_QUERY_TOKENS)` — shape (N,)
6. **Normalization (always dynamic — never hardcoded min/max):**

   ```
   Final Composite = 0.50 × NDCG@10 + 0.30 × NDCG@50 + 0.15 × MAP + 0.05 × P@10
   ```

   BM25 scores are unbounded and relative to the specific document set. Always recompute min/max from the current batch — never assume a fixed range.

**Boundary conditions:**

- Zero-division guard required (if all scores identical, bm25_range=0 → substitute 1.0)
- Disk budget: `du -sh artifacts/` should show <2 GB after precompute Step 4

**Data contract:**

- Output: `bm25_scores_norm` ∈ [0.0, 1.0], shape (100000,), fed to RRF

### Module 5.4 — Career Description Keyword Score

**Output:** `career_keyword_norm` ∈ [0.0, 1.0] → input to RRF fusion

**Logic:**

- Count occurrences of TIER_1_SKILLS keys in concatenated career description text (case-insensitive)
- Apply saturation cap: count is capped at 2 per unique skill key before summing. After the first 1-2 mentions, the relevance signal plateaus; uncapped counting rewards keyword stuffers 8.5x over concise candidates.
- Also guards against false-positive context: "SQL retrieval" counting as a vector retrieval hit — this signal is the weakest of the three; dense embedding handles true semantic disambiguation.
- Normalize to [0, 1] via min-max with zero-division guard (dynamic, never hardcoded range)

**Data contract:**

- Output: `career_keyword_norm` ∈ [0.0, 1.0], shape (100000,), fed to RRF

### Module 5.5 — Dense Embedding Similarity (HyDE)

**Output:** `dense_scores` ∈ [0.0, 1.0] → input to RRF fusion

**Model specification:**

- Model: `sentence-transformers/all-MiniLM-L6-v2`
- Parameters: 22M, Embedding dimension: 384
- Throughput: ~14,000 sentences/second on CPU
- Hard sequence limit: 256 tokens (BERT subword). Text exceeding this is silently truncated.
- Precompute time for 100K candidates: ~7 seconds

**Candidate text construction (`build_candidate_text`):**

Concatenate in this order:

1. `profile.headline` (prepended FIRST — headlines are semantically dense: "ML Engineer specializing in RAG and Vector Search" is more specific than a generic title "Software Engineer II")
2. `profile.current_title`
3. "Skills: " + space-joined skill names where `duration_months > 0`
4. Space-joined career descriptions, **hard-truncated at 300 characters** to stay within 256-token limit
5. `all-MiniLM-L6-v2` silently truncates anything past 256 subword tokens. The order of the string above is strictly structural, not stylistic. The most critical semantic signals (`headline` and `skills`) MUST remain at the very front of the string. The `career_history` text must remain strictly at the tail end so that if token truncation occurs on a candidate with a massive skill list, only historical fluff is lost, preserving the high-value density. Do not reorder these concatenations.

Safe field reads (None-handling):

```
Final Composite = 0.50 × NDCG@10 + 0.30 × NDCG@50 + 0.15 × MAP + 0.05 × P@10
```

⚠️ `dict.get(key, default)` only returns `default` when the key is **absent** — if the key exists with value `None`, you get `None` not the default. Use `or ""` pattern to handle both.

**Hypothetical JD Resume Embedding:**

- Read `hypothetical_resume_selected.txt` (generated in Gate P2)
- Encode with L2-normalization → shape (384,)

**Candidate embedding:**

- Encode all 100K candidate texts with L2-normalization, batch_size=512
- Output: (100000, 384), float32

**Score computation:**

```
Final Composite = 0.50 × NDCG@10 + 0.30 × NDCG@50 + 0.15 × MAP + 0.05 × P@10
```

Since both are L2-normalized, dot product == cosine similarity.

**Rescaling — use `(score + 1) / 2`, not `clip(score, 0, 1)`:**
Cosine similarity ranges [-1, 1]. Clipping to [0, 1] treats a "perfectly opposite" candidate (score -1) identically to an "unrelated" candidate (score ~0). Rescaling `(x + 1) / 2` preserves the full signal: opposite candidate → 0.025, unrelated → 0.510.

In practice, MiniLM rarely produces very negative scores for unrelated text (scores cluster around 0.1-0.4 for unrelated English text), but rescaling is strictly more correct.

**Boundary conditions:**

- Missing `duration_months` on a skill → skill name excluded from text
- Career description truncated at 300 chars (hard truncation, no sentence-boundary respect)

### Module 5.6 — Reciprocal Rank Fusion (RRF)

**Output:** `rrf_scores` ∈ [0.0, 1.0] → features.npy column [0]

**Mathematical Logic:**

Given score arrays S₁, S₂, S₃ (each shape N=100000):

1. For each Sᵢ: compute 1-indexed rank (`rank_i[j]` = position of candidate j when Sᵢ sorted descending; rank 1 = highest score)
2. `rrf_score[j] = Σᵢ (1 / (k + rank_i[j]))` where k=60 (never change this; community standard since 2009)
3. Normalize: `rrf_normalized = (rrf_score - rrf_min) / (rrf_max - rrf_min)` with zero-division guard

**Three inputs fused:**

1. `bm25_scores_norm` — BM25 lexical text scores
2. `dense_scores` — cosine similarity to hypothetical JD resume
3. `career_keyword_norm` — keyword count scores (saturated)

**Why RRF over linear weighted sum:** BM25 scores range 0-20+; cosine scores range 0-1. These scales are incomparable. A linear combination requires careful calibration that shifts with data distribution. RRF discards absolute scores and fuses only rank positions — making scales irrelevant. k=60 is proven optimal and requires no tuning.

**Tie-breaking in argsort:** Use `np.argsort(..., kind='stable')` for deterministic behavior when scores are tied. Default quicksort (introsort) is technically NOT guaranteed stable across NumPy versions/builds.

**Token overlap note:** Several tokens appear in both BM25 query tokens and TIER_1_SKILLS (e.g., "retrieval", "embedding", "ndcg"). These overlapping tokens are counted toward two of the three RRF inputs (BM25 + career_keyword). This is mathematically sound — BM25 and keyword_count capture the same underlying lexical signal through different mechanisms, which the RRF rank-position approach handles correctly (overlapping signals produce correlated ranks, which amplify each other proportionally rather than independently). This is intended behavior, not a flaw.

**Boundary conditions:**

- k=60 (never change without written justification)
- Zero-division guard on normalization denominator
- Output dtype: float32

### Module 5.7 — Skill Gap Coverage

**Output:** `skill_gap_coverage` ∈ {0.0, 0.333, 0.667, 1.0} → features.npy column [2]

**Functional Objective:** Penalize candidates who score high on text retrieval but have zero actual skill-array evidence in core JD-relevant clusters. Prevents text-rich, skill-array-empty candidates from ranking deceptively high.

**Logic:**

1. `trusted_skills` = set of `s.name.lower().strip()` for all skills where `s.duration_months >= 6`
   - Skills with `duration_months < 6` do NOT count as trusted
   - Missing `duration_months` defaults to 0 → not trusted
2. For each of 3 core clusters (embeddings_cluster, retrieval_cluster, vector_db_cluster): check if `trusted_skills ∩ cluster_skills` is non-empty
3. `skill_gap_coverage = (number of covered clusters) / 3`

**Possible values:**

- 1.0 = trusted skill in every cluster → no penalty
- 0.667 = two clusters covered
- 0.333 = one cluster covered
- 0.0 = zero coverage across all clusters → full penalty (controlled by 0.06 weight)

**Data contract:**

- Input: `candidate["skills"]`
- Output: stored in features.npy column [2]

### Module 5.8 — Career Quality Composite

**Output:** `career_score` ∈ [0.0, 1.0] → features.npy column [3]

**Sub-weights (sum to 1.00):**

```
Final Composite = 0.50 × NDCG@10 + 0.30 × NDCG@50 + 0.15 × MAP + 0.05 × P@10
```

Then apply multipliers in sequence (both are post-base multipliers, not sub-weights):

```
Final Composite = 0.50 × NDCG@10 + 0.30 × NDCG@50 + 0.15 × MAP + 0.05 × P@10
```

#### 5.8.1 Company Type Score (Consulting Penalty)

- Consulting firm set: {tata consultancy services, tcs, infosys, wipro, accenture, cognizant, capgemini, hcl technologies, tech mahindra, hexaware, mphasis, ltimindtree, l&t infotech}
- Matching: **Word-boundary regex** `\b{firm}\b` on lowercased company name — prevents "tcs" matching "ATCS Analytics". Escape special chars (e.g., "&" in "l&t infotech").
- `company_type_score = max(product_months / (consulting_months + product_months + 1), 0.02)`
  - Floor 0.02: prevents hard zero (pure consulting gets tiny non-zero score)
  - +1 in denominator: prevents zero-division for empty career histories

#### 5.8.2 Job Stability Score

- If `len(career_history) < 2`: return 1.0
- Otherwise: `short_stints` = count of jobs where `duration_months < 18` AND `is_current == False`
- `stability_score = max(1.0 - (short_stints / len(career_history) × 0.75), 0.1)`
- Floor 0.1 prevents hard zero

#### 5.8.3 Title Relevance Score

For each job, classify title (lowercased) and compute weighted months:

| Category               | Tokens                                                                                                                                                                       | Weight                   |
| ---------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------ |
| STRONG                 | ml engineer, machine learning, ai engineer, applied scientist, nlp engineer, search engineer, ranking, recommendation, retrieval                                             | 1.0× months             |
| RELEVANT               | (all STRONG) + data scientist, research engineer, ranking engineer, backend engineer, software engineer                                                                      | 0.5× months             |
| NEUTRAL_ARCHITECT      | "architect" + any of {software, technical, solution, enterprise, principal} but NONE of {ml, ai, data, search, nlp, cloud, platform}                                         | 0.25× months            |
| DISQUALIFIED           | marketing manager, hr manager, content writer, sales, business analyst, scrum master, project manager, recruiter, computer vision, robotics, speech recognition, cv engineer | 0.0× months             |
| DISQUALIFIED_ARCHITECT | "architect" with no TECH_QUALIFIER and no NEUTRAL_TERM                                                                                                                       | 0.0× months             |
| Unmatched/unknown      | (none of the above)                                                                                                                                                          | 0.0× months, no penalty |

`title_relevance_score = weighted_months / max(total_months, 1)`

Precedence: TECH_ARCHITECT_QUALIFIER takes precedence. If "architect" + "ml" → NOT disqualified → check STRONG/RELEVANT.

#### 5.8.4 Startup Trajectory Score (Schema-Driven)

**Replaces the old name-marker heuristic** (which guessed company size from strings like "labs", "tech", "io"). The schema provides `company_size` on `career_history[].company_size` and `profile.current_company_size`. This is a direct, schema-backed signal.

**JD connection:** "growing the team from 4 to 12 engineers" = startup-scaling experience is an explicit positive signal.

### Module 5.9 — The Three JD Disqualifier Traps

The JD explicitly names six disqualifying patterns. Three are already covered elsewhere (keyword stuffers by duration gate, consulting careers by company_type_score, ghost profiles by honeypot detection). These three require dedicated logic:

#### 5.9.1 LangChain-Only Trap

> JD: "If your 'AI experience' consists primarily of recent (under 12 months) projects using LangChain... we will probably not move forward."

Targets candidates who wrapped a thin LangChain integration around an LLM API call with no underlying retrieval/embedding skill evidence.

**Multiplier applied to `skill_match_score` (immediately after base calculation):**

Pattern conditions (ALL must be true):

- Has "langchain" in skills array
- Has ZERO skills from RETRIEVAL_EVIDENCE_SKILLS = {rag, retrieval augmented generation, pinecone, faiss, milvus, qdrant, weaviate, embeddings, embedding, vector database, vector db, vector store, hybrid search, hybrid retrieval}
- Career only began from 2023 onward (earliest job start_date year ≥ 2023) — meaning LangChain is plausibly their only AI-adjacent experience

If all conditions true → multiplier = 0.5. Otherwise → 1.0 (no penalty).

Fallback: if no career history data → return 1.0 (cannot evaluate the "recent only" condition; no penalty).

#### 5.9.2 CV / Speech / Robotics Domain-Mismatch Trap

> JD: "People whose primary expertise is computer vision, speech, or robotics without significant NLP/IR exposure."

**Multiplier applied to `career_score` (after industry_relevance, before final clamp):**

```
Final Composite = 0.50 × NDCG@10 + 0.30 × NDCG@50 + 0.15 × MAP + 0.05 × P@10
```

`cv_count` = number of candidate skills (by name, lowercased) in CV_SPEECH_ROBOTICS_SKILLS

`nlp_count` = number of candidate skills in NLP_IR_EVIDENCE_SKILLS

If `cv_count >= 5` AND `nlp_count == 0` → multiplier = 0.3. Otherwise → 1.0.

Threshold of 5 avoids penalizing candidates who list "ROS" once in a broader ML skill set — specifically targets skill arrays dominated by the wrong domain.

> JD: "Senior engineer who hasn't written production code in 18 months because you've moved into 'architecture' or 'tech lead' roles."

**Implemented directly inside `behavioral_multiplier()`** (not as a standalone function — fundamentally a recency/activity signal).

Conditions (BOTH must be true):

- `current_title` (lowercased) contains any of: "architect", "tech lead", "principal"
- `days_inactive > 540`

**Detection Rules (37 total rules defined — 36 active, 1 removed — ANY active match flags as honeypot):**

## SPEC-7 — Reasoning Engine

### 7.1 Architecture Constraint

**Single path. Zero network calls. Fully procedural. Deterministic.**

The LLM is used ONLY in Gate P2 (offline manual step, before precompute). All reasoning text is generated by procedural fact extractors. No API key or network calls are required in the codebase. `rank.py` MUST NOT import `anthropic`, `openai`, or any network library. Verify: `grep -r "anthropic" src/rank.py` → must return no matches.

### 7.2 Fact Extractor Priority Queue (12 extractors)

| #  | Extractor          | Example Output                             | Type                    |
| -- | ------------------ | ------------------------------------------ | ----------------------- |
| 1  | experience_fact    | "7.2y applied ML background"               | Positive                |
| 2  | title_fact         | "current role as ML Engineer at [company]" | Positive                |
| 3  | top_skill_fact     | "Milvus (36mo, assessed 78/100)"           | Positive                |
| 4  | company_type_fact  | "product-company career at [companies]"    | Positive                |
| 5  | retrieval_fact     | "career text mentions embedding pipelines" | Positive                |
| 6  | github_fact        | "active open-source (GitHub: 72/100)"      | Positive                |
| 7  | response_rate_fact | "recruiter response rate: 84%"             | Positive                |
| 8  | location_fact      | "Pune-based, no relocation needed"         | Positive                |
| 9  | notice_fact        | "15-day notice period"                     | Positive                |
| 10 | active_fact        | "active on platform (last seen: 3d ago)"   | Positive                |
| 11 | education_fact     | "tier_1 institution"                       | Positive                |
| 12 | concern_fact       | "notice period 90d is above ideal"         | Negative (always fires) |

### 7.3 top_skill_fact Logic

1. Filter candidate["skills"] to JD-relevant skills (name in JD_TAXONOMY) where `duration_months >= 6`
2. If no relevant trusted skills → return None
3. Score: `base × min(duration/24, 1.0) × (assessment_score/100 or 0.65)`
4. Select highest-scoring skill
5. Build text: `"{skill_name} ({duration_months}mo, assessed {score}/100)"` — omit assessment if not in redrob_signals.skill_assessment_scores
6. ⚠️ Assessment scores: read from `redrob_signals.skill_assessment_scores` — NOT from the skills array

### 7.4 concern_fact Logic

Priority order (highest severity first):

- `days_inactive > 60` → "inactive {days}d on platform"
- `notice_period_days > 60` → "notice period {days}d"
- `recruiter_response_rate < 0.3` → "low recruiter response rate ({int(rr*100)}%)"
- `interview_completion_rate < 0.5` → "interview completion rate {int(icr*100)}%"
- `verified_email == False` → "email unverified"
- Fallback (always fires): "no critical concerns identified"

All signal reads use `.get(key, default)`. Uses `TODAY` loaded from `artifacts/dataset_today.txt` via `src/constants.py` (SPEC-2.2 — dataset-derived `DATASET_TODAY`), never `date.today()`, never a hardcoded calendar date.

### 7.5 Reasoning Assembly

1. Run all 12 extractors. Collect non-None results.
2. Sort by score descending.
3. Select top 2 positive facts.
4. Select top concern (always present).
5. Apply rank-based tone prefix.
6. Format: `"{prefix} {fact1}; {fact2}. Concern: {concern}."`

### 7.6 Tone Prefix Table

| Rank Range | Prefix                       |
| ---------- | ---------------------------- |
| 1-5        | "Top fit:"                   |
| 6-20       | "Strong match:"              |
| 21-50      | "Solid candidate:"           |
| 51-80      | "Marginal fit:"              |
| 81-100     | "Below cutoff but included:" |

### 7.7 Reasoning Constraints

- **Allowed:** `"6.8 years experience"`, `"Milvus (24 months)"`, `"Response rate 82%"` — specific facts from the profile
- **Forbidden:** `"Strong leadership"`, `"Excellent communicator"` — claims without profile evidence
- **Anti-hallucination gate:** Every capitalized technical word in reasoning must exist in candidate's skills array. Whitelist for common acronyms: {ml, ai, nlp, rag, sql, aws, gcp}. This check scans **ALL 100 rows** (not just top 20) — Stage 4 samples 10 RANDOM rows from the full 100.
- **Target length:** 1-2 sentences, 25-60 words
- **All strings through `csv.writer`** — handles commas, quotes automatically

## SPEC-16 — Stage 5 Defense Answers

### Architecture Questions

| Question                                                 | Answer                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| -------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Why not cross-encoder?                                   | CPU-only, 5-min constraint. Cross-encoder on 100K = ~0.1s/candidate = 10,000 seconds. MiniLM bi-encoder precomputes in ~7s; cosine similarity at runtime is a single matrix multiply.                                                                                                                                                                                                                                                                                                                                                          |
| Why not LightGBM?                                        | No real training labels. Training on our own heuristic scores is circular — it learns to reproduce our heuristic, not the ground truth. Weighted scoring is interpretable and debuggable.                                                                                                                                                                                                                                                                                                                                                     |
| Why multiplicative behavioral multiplier?                | An unreachable candidate is worthless. Multiplicative means behavioral veto is possible: 0.25× on a strong profile → low final score. Additive would let 100% response rate compensate for zero embedding experience.                                                                                                                                                                                                                                                                                                                        |
| Why is duration_months the primary anti-stuffing signal? | Self-declared proficiency is unverifiable. Assessment scores exist only for a subset. Duration is the only cross-validated signal confirming actual skill usage in a real job context.                                                                                                                                                                                                                                                                                                                                                         |
| Why exclude salary signal?                               | No explicit budget in JD. "Competitive comp" is not a number. Using salary introduces unintended bias against high-caliber negotiable candidates.                                                                                                                                                                                                                                                                                                                                                                                              |
| How does honeypot detection work?                        | 36 active rules across structural impossibilities (timeline contradictions, duration math, schema bounds), engagement-fraud fingerprints (uniform skill distribution, views/saves paradox, mass applier bots), and ghost profiles. Two rules (Future Cert, Negative Tenure on current job) use the dataset-derived`DATASET_TODAY` constant for deterministic reproducibility — see below.                                                                                                                                                   |
| Why startup_trajectory_score instead of name markers?    | The schema provides`company_size` on every career_history entry — a direct, schema-backed signal. Name-marker heuristics (guessing "startup" from "labs" in company name) are noisy. Direct data is always preferable.                                                                                                                                                                                                                                                                                                                      |
| Why industry_relevance uses career_history[].industry?   | It was an unused schema field. Candidates whose entire career is in AI/ML/Search/HR-Tech industries are higher domain fits than those in Manufacturing or Finance, independent of their job titles.                                                                                                                                                                                                                                                                                                                                            |
| Why is Bangalore excluded from Tier 2?                   | The JD does not mention Bangalore as a target location. Including it created geographic bias away from the JD's stated Pune/NCR requirement.                                                                                                                                                                                                                                                                                                                                                                                                   |
| Why was Honeypot Rule 14 (Endorsement Physics) removed?  | It assumed`endorsements_received` (total skill endorsements, any source) is bounded by `connection_count` (Redrob-specific connections). These fields are not related in the schema — a candidate can legitimately have far more endorsements than Redrob connections via LinkedIn imports or external referrals. Enforcing the rule would have flagged real strong candidates as honeypots, directly hurting NDCG. The fraud pattern it targeted is already caught with lower false-positive risk by Rules 6, 26, and 32 in combination. |

### Research-Backed Questions

| Question                                                                         | Answer                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| -------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Why HyDE over JD-direct embedding?                                               | CONFIT V2 (ACL Findings 2025): +22% nDCG from embedding an LLM-generated ideal candidate profile vs. embedding the JD text directly. The model's embedding space aligns resumes to resumes far better than JDs to resumes. We generated three variations via Claude API, selected the best by cosine similarity against 10 known-good candidates from the full 100K dataset (not the sample bundle), and committed the winner to the repo.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| Why RRF over linear combination?                                                 | BM25 scores (0-20+) and cosine scores (0-1) are incomparable scales. Fixed linear combination requires calibration that shifts with data distribution. RRF (Cormack, Clarke & Buettcher 2009) is scale-invariant, parameter-free (k=60), proven optimal across retrieval benchmarks. OpenSearch 2024 across 6 datasets: RRF consistently outperforms both BM25-alone and dense-alone baselines.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| Why is this system v3, not v2?                                                   | JD describes v2 = embeddings + hybrid retrieval + LLM re-ranking. This system adds beyond v2: (1) hypothetical resume embedding (HyDE) instead of JD-direct embedding, (2) RRF fusion instead of linear combination, (3) skill-gap coverage vector preventing text-strong/skill-array-empty false positives, (4) startup_trajectory_score using actual schema company_size field, (5) industry_relevance multiplier using career_history[].industry, (6) three explicit JD disqualifier penalties not present in any v2 system description.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| Why determinism via dataset-derived anchor vs. date.today() or a hardcoded date? | `date.today()` produces different rankings depending on when code executes — a candidate's `days_inactive` grows every day rank.py is re-run, silently shifting scores between identical runs. A hardcoded calendar date solves that but introduces a worse failure: once real evaluation happens after the hardcoded date, every candidate's `days_inactive` is inflated by the same growing offset, and the entire pool eventually collapses into the harshest inactivity penalty tier together — destroying the signal exactly when Stage 3 reads it. The correct fix derives `TODAY` from the dataset itself: the most recent `last_active_date` across the candidate pool, computed once in `precompute.py`, persisted to `artifacts/dataset_today.txt`, and loaded by both `precompute.py` and `rank.py`. Re-running the identical dataset always produces the identical `TODAY` — true determinism — while a genuinely different dataset (Stage 3's hidden pool) correctly produces a different, equally valid `TODAY` anchored to that pool's own timeline. This is what `deterministic_mode: true` actually requires: reproducibility for a given input, not a frozen constant that goes stale. |

### profile Fields (all required)

`anonymized_name`, `headline`, `summary`, `location`, `country`, `years_of_experience` (number 0-50), `current_title`, `current_company`, `current_company_size` (enum: "1-10" / "11-50" / "51-200" / "201-500" / "501-1000" / "1001-5000" / "5001-10000" / "10001+"), `current_industry`
